[Feynman Lectures on Computation], sec.6.2.

It is sometimes helpful to look at computational circuits not in terms of logic gates per se but in terms of **reversible primitives**, such as the **controlled not** (i.e., the b bit flips only if a=1). Viewed in terms of reversible primitives, an adder can be thought of as three wires. The third write is originally 0, but if a=1 and b=1, it is flipped to 1, otherwise, it stays 0; therefore, the third wire is the **carry**. The second wire originally holds the value of b, which flips when a=1 and the current flows left to right; therefore, the second wire is the **sum**. Controlled nots can also be arranged in such a way that we preserve the values of a and b on their wires and store the sum on a separate wire.

## Budgeting transistors across CPU cores and caches

> In 1974, the Intel 8080 contained a little over 2000 transistors, while Xeon Nehalem-EX CPUs have over 2 billion transistors. An obvious question is: ‘‘What do you do with all those transistors?’’ ... One option is to add megabytes of cache to the chip. This option is serious, and chips with 4–32 MB of on-chip cache are common. But at some point increasing the cache size may run the hit rate up only from 99% to 99.5%, which does not improve application performance much. The other option is to put two or more complete CPUs, usually called cores, on the same chip.

[Modern Operating Systems], sec.8.1.1.

Suppose we were redesigning a CPU with the goal of adding more cache memory to it. Due to physical constraints, there's a fixed number of transistors we can have on a chip, so to add a cache, we need to remove another component, e.g., a CPU core. If 6 transistors are needed to store 1 bit of cache and removing a core freed up 200 million transistors, then we could use those to add 4 MB of cache.

## When adding too many CPU cores hits the coherency wall

> The machinery needed to keep their caches coherent becomes very complicated and very expensive. Many engineers worry that cache coherence may not scale to many hundreds of cores. Some even advocate that we should give it up altogether. They fear that the cost of coherence protocols in hardware will be so high that all those shiny new cores will not help performance much because the processor is too busy keeping the caches in a consistent state. Worse, it would need to spend way too much memory on the (fast) directory to do so. ... Some engineers argue that the only programming model that has proven to scale to very large numbers of processors is that which employs message passing and distributed memory.

[Modern Operating Systems], sec.8.1.1.

Adding CPU cores could improve performance... until we hit the cache coherency wall. At a certain point, keeping caches consistent consumes so many CPU cycles (more than the parallelism saves) that a program may actually run slower than had it run on fewer cores.

## Apple M1: how to pass performance benchmarks while also boasting about a long battery life

> A CPU can have a small number of ‘‘big’’ cores, with deep pipelines and possibly high clock speeds, and a larger number of ‘‘little’’ cores that are simpler, less powerful, and perhaps run at lower frequencies. The powerful cores are needed for running code that requires fast sequential processing while the little cores are useful for power-efficiency and for tasks that can be executed efficiently in parallel. ... The M1 chip, used on some Apple computers and mobile devices, is an SoC with four high-performance, power-hungry cores and four lower-performance, energy-efficient cores. This gives the operating system the ability to run threads on fast cores when that is needed but save energy when it is not.

[Modern Operating Systems], sec.8.1.1.

SoC = System On a Chip. The M1 is a heterogeneous processor designed to conserve power but also deliver high performance when needed. This is how the modern MacBook Pro has such high performance benchmarking scores and also long battery life.
